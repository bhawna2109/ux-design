Find and replace

Designed the experience for performing bulk refactor in Postman



This story highlights the importance of trimming down a design.

Problem

Postman did not support an app-wide find and replace functionality

Users work with large amounts of data in Postman. However, there was no support to find something across all or parts of that data. Furthermore, there was no replace functionality. Thus, if a variable name got changed, users had to manually replace each instance of the variable across the app. This was time consuming and not productive.

Find and replace was a long awaited feature in Postman - we had been getting several feature requests on Github over the years around supporting bulk refactoring in the app.

The challenge

How might we support finding and replacing text in Postman?

Our goal for the project was to support finding and replacing text across all data in Postman. The higher level goal was to support quick search and reduce the time that users spent in replacing things one by one, thus providing productivity gains to users.

Responsibilities

My role

I was the sole designer on this feature, and worked in close collaboration with one developer.

As the only designer working on this feature, my responsibilities included:

  • Understanding the underlying need of the users
  • Learning about bulk refactoring in code
  • Wireframing: Understanding the Postman interface and creating wireframes to identify the best real estate to introduce this feature
  • Creating high-fidelity mocks and prototypes for the final solution
  • Working closely with engineers to ensure high-quality implementation
  • Writing documentation

Solution

Introducing Find and Replace in Postman

Find anything across the Postman app

To find something, simply click on the “Find” icon in the status bar or hit ‘CMD+SHIFT+f’ on Mac or ‘CTRL + SHIFT + f’ on Windows and Linux devices. Enter your search query and hit 'Enter' or click on the Find button.

The process

Understanding the 'Why', 'Who', 'When' and 'Where'

To start off, I wanted to deeply understand the goal (Why), the users (Who), and the needs and context of the users (When and Where). I started off by delving deep into the feature requests around find and replace on github and grouped them in a single project. Next, to understand their needs and context, I wrote Job Stories from the feature requests on github.

User persona

Understanding the preferences of primary users of Postman

The primary users of Postman are developers. After getting a rough idea of user needs, and before starting to design, I wanted to write down the design principles which I would base the design on. I derived the design principles from user preferences.

What do developers like and dislike? Talking to my developer peers at Postman, and interacting with developers through github, twitter and zendesk helped me understand some likes and dislikes of developers.

Laying down the design principles

  • Make workflows more efficient by supporting keyboard based workflows
  • Focus on how quickly and effectively users are able to get their job done
  • Show as much information upfront as possible - don't limit what the user wants to see
  • Eliminate interaction by inferring context (by environment, history of past environments and interactions). Inspiration: Bret Victor (http://worrydream.com/MagicInk/)
    • The best way to eliminate interaction is through information rich graphic design that uses the environment and history
    • Interaction modes effectiveness: mouse < touch < speech < sheer amount of information that can be absorbed by the eye
  • As far as possible, provide information without changing the context or interrupting the workflow (inline popups etc.)
  • Optimize UX for fast loading times whenever possible

Comparative analysis

What works and does not work in comparative developer products?

I analysed other developer products which have the find and replace functionality to understand:

  • Any existing mental models that developers might have while using find and replace
  • Good design elements and interactions
  • Get an overall idea about tasks involved in finding and replacing

Solving

Defining tasks

To start wireframing, I listed the tasks that a user would need to be able to complete in order to find and replace something successfully.

  • Launch find and replace
  • Enter "what" to find"
  • Configure "where" to find
  • View search results
  • Open each search result in builder
  • Enter text to replace with
  • Choose search results to replace
  • Exit find and replace

Wireframing

Early explorations

To start off, we firstly wanted to identify the real estate in the app where the find and replace feature would exist. After some intial brainstorming, we decided to have a launch point in the bottom bar which would bring up the find and replace bottom sheet. I started sketching rough flows and then created first draft of wireframes. These wireframes went through a design-dogfooding-iteration loop before we converged on the final design.

Bringing up the find and replace bottom sheet

We found from comparative analysis that CMD/CTRL+SHIFT+F is an expected shortcut to bring up find and replace (all) across all developer tools. To preserve user's existing mental model, we decided on having the same shortcut to bring up the "Find and replace" bottom sheet. We also added "Esc" key for exiting find and replace.

In addition to a keyboard shortcut, we also added a prominent find and replace icon in the bottom bar. Clicking this icon would open up a bottom sheet to perform find and replace. We intentionally decided to not make this bottom sheet a modal in order to NOT make the rest of the interface non usable.

Deciding the layout based on the most important tasks

I explored a few different layouts to help the user complete the most important tasks. We decided to put function over aesthetics following our design principle "Show as much information upfront as possible - don't limit what the user wants to see" and "As far as possible, provide information without changing the context or interrupting the workflow"

Find "what"

To solve for this task, I included a text box to enter the search query. In addition, I added the options "Regex" and "Ignore case" to allow for more flexible searches (implementing our finding from comparative analysis).

Our comparative analysis also showed that some tools support real-time search which is a good-to-have. However, there were implementation challenges associated with showing search results as the user was typing. Thus, we decided to show search results only when an explicit "Find" action was triggered by the user.

Find "where"

From our job stories, we know that users wanted the ability to find within specific Postman entities (current collection vs all collections). In addition, our job stories also show that users wanted to search within specific "fields" within an entity, for example, "URL field" of a request or "tests field" of a request. It was clear that we needed to support the ability to view search results for specific Postman entities (collections, environments, globals), as well as specific fields within those entities. I noted down all the Postman entities and the fields within each entity (which were different for each entity).

Design rationale behind the final design decision

I created an internal team wiki page (on Confluence) and wrote down the design rationale behind the design decision of supporting filtering action on fields after the search results are shown.

  • Scalability: Showing both entities and fields before searching makes the Find section overwhelming. This complexity will increase when we add more entities/fields in the future.
  • User context: From a user's perspective, they're almost always sure about the entity they want to find in. Our job stories show that typically they're working with a single collection and want to perform find/replace within that. In addition, the job stories also listed few scenarios when users did NOT specify a "field" to search in. This design supports those scenarios while also supporting the ability to filter/play around with the search results later.
  • Fast loading: If a user chooses specific entities instead of finding in "everything", they'll get the search results faster, since there will be less data to perform the search in the backend. We will not get the same loading advantage if a user could limit the "fields" too before search. Supports our design principle "Optimize UX for fast loading times whenever possible."
  • Quick results: This approach ensures that the user does not have to set up a proper configuration before seeing a single search result. Supports our design principle "Focus on how quickly and effectively users are able to get their job done."
  • Better context: Since fields are different for each entity, they can be shown contexually for each entity (tab) in search results.

Final designs

Design handoff and launch

I created the user flows on sketch and paired with a developer who started building a prototype. We worked closely to ensure pixel perfect implementation. This feature is now fully launched for all Postman users.

Testing

How is the find and replace feature being used?

As part of the design process, the designers had to lay out the user interactions that were valuable to track in order to measure success as well as identify areas of improvement. As the data started coming in, we used Microsoft's Power BI to analyse and make sense of the data (by working closely with the data science team).

Future work

What can be improved?

  • Accessibility: Having more color contrast for better readability
  • Improving UX for the "fields" dropdown. Having a sizeable amount of fields in the dropdown list can make browsing harder. In addition, having an option to select or deselect will help if a user wants to search within only one field.
  • Look at user analytics data to see which entities the users are searching in as well as how the field filters are being used. I believe there is more opportunity to simplify the design by removing options that are not quite useful.

Core learning

Biggest learning from this project

This project helped me learn the importance of 'trimming down' a design. I learned that adding interface elements is easy, but taking them off is hard. Thus, it is very important to utilise the real estate judiciously and keep ONLY the absolutely required interface elements for a cleaner and minimalistic design that 'works'.
